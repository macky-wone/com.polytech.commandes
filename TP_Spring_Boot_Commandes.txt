Application de Gestion de Commandes

Contexte pédagogique
Ce Travail Pratique s’inscrit dans le cadre du module Développement d’Applications 
Backend avec Spring Boot (niveau cycle ingénieur / école polytechnique).

L’objectif est de concevoir une API REST professionnelle, conforme aux standards 
industriels, intégrant :
- une architecture en couches,
- la persistance avec JPA,
- l’initialisation contrôlée des données,
- la documentation OpenAPI,
- la gestion des profils Spring et des variables d’environnement.

Objectifs pédagogiques
À l’issue de ce TP, l’étudiant devra être capable de :
• Concevoir une application Spring Boot structurée et maintenable
• Implémenter un CRUD complet avec Spring Data JPA
• Appliquer une séparation stricte des responsabilités (Controller / Service / Repository)
• Externaliser la configuration via Spring Profiles
• Utiliser des variables d’environnement pour les paramètres sensibles
• Documenter une API REST avec OpenAPI / Swagger
• Préparer une application déployable en production

Contexte métier
Une entreprise souhaite mettre en place un système de gestion de commandes 
clients exposé sous forme d’API REST afin de permettre une future intégration avec 
des applications web ou mobiles.

Le système doit permettre :
- la gestion des produits,
- la gestion des clients,
- la création et le suivi des commandes,
- une documentation API claire et exploitable.

Architecture imposée
L’architecture suivante est obligatoire :

com.polytech.commandes
│
├── entity
├── repository
├── service
│   └── impl
├── controller
├── dto
├── exception
├── config
└── CommandesApplication.java

Règles impératives :
- Aucune logique métier dans les contrôleurs
- Les entités ne doivent pas être exposées directement (utilisation de DTO)
- Chaque entité doit disposer d’un Repository, d’un Service et d’un Controller

Modèle de données

Produit
• id : Long
• nom : String
• prix : BigDecimal
• stock : Integer

Client
• id : Long
• nom : String
• email : String

Commande
• id : Long
• dateCommande : LocalDateTime
• status : enum (CREATED, VALIDATED, CANCELLED)
• client : Client
• lignes : List

LigneCommande
• id : Long
• produit : Produit
• quantite : Integer
• prixUnitaire : BigDecimal

Relations JPA attendues
• Client 1..N Commande
• Commande 1..N LigneCommande
• Produit 1..N LigneCommande

La gestion correcte des annotations JPA (@OneToMany, @ManyToOne, cascade, fetch) 
et de la sérialisation JSON est obligatoire.

Travail demandé

Partie 1 – Couche Repository
• Création des repositories JPA
• Héritage de JpaRepository
• Au moins deux méthodes de recherche métier

Partie 2 – Couche Service
• Interfaces + implémentations
• Implémentation des règles métier suivantes :
  o interdiction d’une commande si le stock est insuffisant,
  o calcul automatique du total de la commande,
  o mise à jour du stock après validation.

Partie 3 – API REST
Chaque ressource doit exposer des méthodes pour faire du CRUD.
Les codes HTTP doivent être conformes aux standards REST.

Profils Spring et configuration
L’application doit supporter trois profils :

Fichiers attendus
application.yml
application-dev.yml
application-test.yml
application-prod.yml

Variables d’environnement obligatoires
Aucune information sensible ne doit être codée en dur.

Variables exigées :
- SPRING_PROFILES_ACTIVE
- DB_URL
- DB_USERNAME
- DB_PASSWORD
- SERVER_PORT

Profil / Usage
dev : Développement local
test : Tests automatisés
prod : Production

Initialisation des données
Un CommandLineRunner doit :
- insérer des clients,
- insérer des produits,
- créer au moins une commande complète.

Ce composant doit être actif uniquement sur le profil dev.

Documentation OpenAPI
L’API doit être documentée avec OpenAPI :
- Swagger UI accessible
- Description claire des endpoints
- Regroupement par ressource (tags)
- Mention du profil actif et des prérequis de configuration
